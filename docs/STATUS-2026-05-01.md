### Foundation (already in place)

- Three-pack architecture: `SpeakerPack × CoursePack × JurisdictionPack` per `types/packs.ts`
- Bundled-lesson loading via `getCourseLesson(courseId, speakerId, lessonId)` — synchronous, no network
- Entitlement gating in `lib/entitlements.ts` — DEV_UNLOCK_ALL bypasses everything for the developer
- Supabase Storage bucket `course-content/lessons/...` — 500 JSONs uploaded, **not currently read by app** (future hot-update channel)
- i18n via `lib/i18n.ts` — i18next loads 25 locale files keyed by ISO code (en/sn/fr/zh/tl)

## Open issues — must fix before tester distribution

### 🔴 Rwen AI interface is broken

User reports the Rwen AI chat / interaction screen is "totally messed up at the moment" — needs full diagnosis and fix. This is the second-order experience for users who hit Phase 8 lesson conversations and the AI Companion course; if it's broken, paying tiers can't deliver value. **Top priority before sending to testers.**

Likely surfaces to check:
- `app/lesson/[id].tsx` Phase 8 dialogue branch
- AI Companion screens (search for `ai-companion` course usage)
- `lib/claude.ts` system prompt assembly + token substitution from new speaker packs (especially the new fr/zh/tl packs that just landed — verify their `aiSystemPrompt.contextTemplate` tokens render correctly)
- `lib/elevenlabs*` voice integration
- ElevenLabs API key, voice IDs, latency

### 🟡 Native review of `__warnings` arrays

Every new speaker pack JSON has a `__warnings` array flagging strings that need native-speaker eyes. Inert at runtime so the app ships fine, but the legal/consent strings in `auth.json` particularly need a verified native pass before public launch. Schedulable as a separate work block — not a blocker for closed testing.

### 🟡 Bundle size

Now bundling all 500 lesson JSONs + 5 speaker packs into the JS binary. Worth checking the `npx expo export` output to confirm size is acceptable — large bundles slow first-launch. The synchronous `require()` in each course's `getLessonSync` likely defeats the lazy `curriculumLoader` code-splitting that's already wired.

### 🟢 DB-driven content (deferred)

Storage uploads exist but the app reads bundled. **Don't switch before testing** — adds offline/cache complexity. Wire DB-pulling for v1.1 as a hot-update channel (fix typos without app store release).

## Steps to launch (in order)

### Phase A — Personal smoke test (TODAY, in progress)

1. ✅ `EXPO_PUBLIC_DEV_UNLOCK_ALL=1` added to `.env.local` — bypasses every paywall and module-5+ lock for the developer
2. ✅ `npx expo start --tunnel` running in background — scan QR with Expo Go on phone
3. **You:** Wipe old test accounts via Supabase SQL editor or ask me to cascade-delete via MCP
4. **You:** Smoke-test the unlocked flow — every course, every module, every lesson type. Note what's broken.
5. **Critical:** Rwen AI interface — reproduce the "totally messed up" bug, capture screenshots / console errors

### Phase B — Tester-build setup (1-2 days)

Required before sending to non-technical testers:

1. **Decide platforms** — iOS, Android, or both?
2. **Apple Developer account** ($99/yr) — required for TestFlight, lead time ~24h for approval
3. **Google Play Console** ($25 one-time) — required for Internal Testing, faster onboarding
4. **EAS account + `eas.json` config** — currently no `eas.json` exists; need to create one with a `preview` profile that:
   - Strips `EXPO_PUBLIC_DEV_UNLOCK_ALL` (testers must hit the locked flow)
   - Points at production Supabase (or a staging project if you spin one up)
5. **First EAS build** — `eas build --profile preview --platform <ios|android|all>` (~20-30 min per platform on free tier)
6. **TestFlight upload** (iOS) and/or **Internal Testing track** (Android)

### Phase C — Tester distribution (the week after Phase B)

1. Invite testers via email through TestFlight / Play Internal Testing console
2. Testers install via TestFlight app or Play Store internal link
3. Each tester runs through their starter course (modules 1-4 free), hits paywall on module 5, sees the Phase 8 AI gate
4. Collect feedback via TestFlight feedback (iOS) / Play Console (Android) / a shared form
5. Watch for crashes via EAS Insights / Sentry (if wired)

### Phase D — App Store submission (after tester signal)

Out of scope for this snapshot — a separate planning block.
