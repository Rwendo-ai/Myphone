**Status:** Planning / Pre-build
**Decision date:** 2026-05-16
**Owner:** Rwendo core
**Companion to:** `companion-animation.md` (to be written), `elevenlabs.md`, `elevenlabs-conversational.md`

---

## 1. What this is, in one paragraph

The viseme atlas is Rwendo's **default-tier (free)** lip-sync engine. For each 2D photo-realistic companion, we pre-generate a small bundle of ~20 mouth-region images (the "atlas") using a high-quality AI lip-sync model **once, offline, at companion-creation time**. At runtime the client plays back TTS audio from ElevenLabs and composites the correct mouth frame onto a stable base portrait, driven by character/phoneme timestamps that ElevenLabs returns alongside the audio. Result: photo-realistic mouth motion, zero GPU at runtime, zero per-stream cost, works offline, ships on any phone.

> **Important framing — the atlas does NOT generate or store any voice.**
> The voice is always live ElevenLabs TTS / Agents output. The atlas is a *passive renderer* that reacts to whatever audio ElevenLabs is producing — picking the right mouth shape from the pre-baked bundle based on EL's character timestamps and compositing it over the still portrait. Think of the atlas as a teleprompter for the face, not as a voice clone. If ElevenLabs goes down, the atlas stops moving but the photo still shows; if the atlas fails to load, ElevenLabs still talks but the face doesn't animate. Loose coupling, separable failure modes.

Conceptually adjacent to Talking Tom only in the sprite-region compositing trick — *not* in the voice loop. Talking Tom captured the user's voice and replayed it through a cartoon. We do the opposite: a real photo-realistic face whose mouth animates in time with an ElevenLabs AI voice.

The premium tier (live MuseTalk/Hallo3 streaming for full head motion and expression) is a **separate** engine, documented in `lipsync-live.md` (to be written). This doc covers only the free / default tier.

## 2. Decision summary

### What we are building
- A **viseme atlas asset bundle** format (base portrait + ~20 mouth-region crops + metadata)
- A **server-side generation pipeline** that creates one atlas per companion (one-time job)
- A **client-side runtime** that composites the right mouth frame onto the base portrait in sync with ElevenLabs audio
- Ambient micro-animations (blink, breath, head sway) layered on top, all client-side
- Integration into the existing Rwendo companion-selection flow

### What we are not building (here)
- Real-time AI lip-sync (that's the premium tier — separate doc)
- Animation for Rwen the chameleon and other 3D characters (covered by Three.js morph-target system — separate doc)
- Voice cloning or TTS (already in stack via ElevenLabs)
- Phoneme/STT inference (we use ElevenLabs character timestamps as the source of truth)

### What this replaces / removes
- **Removes the need for Pipecat** for the default tier (free users)
- **Removes the need for Daily.co WebRTC** for the default tier
- **Removes Deepgram entirely** (ElevenLabs Scribe inside Agents handles STT)
- Restores the 2026-04-30 architecture decision: ElevenLabs Agents handles the voice loop end-to-end; rendering is purely client-side

### Why this is the right call
1. **Unit economics:** ~$0 per stream-minute at runtime. Atlas generation is a ~$0.01–$2 one-time cost per companion depending on which generator we use.
2. **Quality:** photo-realistic mouth motion. Identity perfectly preserved (base portrait is the user's actual photo). Mouth shapes generated by SOTA AI model.
3. **Reach:** runs on cheap phones, works offline, no server roundtrip per frame.
4. **Premium upsell path:** users who want live full-motion lip-sync can pay for the premium tier later. The atlas tier sets the baseline.
5. **Product fit:** companion sits face-to-face with the user. Limited head motion is a feature, not a bug.

## 3. How the viseme atlas works

### The core insight
A speaking face's mouth cycles through a small, finite set of distinct shapes called **visemes**. Real speech blends them continuously, but a viewer perceives correct lip-sync if the right shape appears at the right time, with a smooth transition. Modern AI lip-sync models generate these shapes fresh every frame — expensive. The atlas approach pre-generates them once and swaps them at runtime — free.

### The mouth-region trick
Don't swap the **whole face** between frames — only the **mouth region**. The base portrait stays stable for everything except the mouth (~10% of pixel area). Lighting, skin grain, hair, eyes all stay rock-stable. The mouth animates. This is the difference between "puppet-like" and "convincing."

### Composition at runtime
```
Base portrait (one PNG, ~500 KB)
    +
Mouth crop for current viseme (one PNG from atlas, ~50–150 KB)
    overlaid at landmark-defined mouth position
    with alpha-matte edges for seamless blending
    =
Final rendered frame
```

Composited on the client at 60fps. Negligible CPU/GPU cost.

## 4. The viseme set

We use a **20-frame atlas** structured as:

### 4.1 Core visemes (12)
The minimum set of distinct mouth shapes. Loosely based on Preston Blair + extensions.

| ID | Name | Example phonemes | Mouth description |
|----|------|------------------|-------------------|
| `sil` | Silence / rest | (no speech) | Closed, neutral |
| `pp` | Bilabial | P, B, M | Lips pressed together |
| `ff` | Labiodental | F, V | Upper teeth on lower lip |
| `th` | Dental | TH, DH | Tongue tip visible between teeth |
| `dd` | Alveolar | T, D, S, Z, N, L | Slight open, tongue near alveolar ridge |
| `kk` | Velar | K, G, NG | Open mid, tongue back |
| `ch` | Postalveolar | CH, J, SH, ZH | Rounded slight pucker |
| `aa` | Open vowel | AH, AA | Wide open mouth |
| `e` | Mid front vowel | EH, AE, AY | Mid-open, lips relaxed |
| `i` | Close front vowel | IY, IH | Slight smile, teeth visible |
| `o` | Mid back vowel | OW, AO | Rounded, medium open |
| `u` | Close back vowel | UW, UH | Tight pucker |

### 4.2 Transition frames (3)
Smoother in-between shapes that bridge common viseme pairs.

| ID | Description |
|----|-------------|
| `t_closed_open` | Mid-position between `pp` and `aa` |
| `t_round_open` | Mid-position between `o` and `aa` |
| `t_smile_open` | Mid-position between `i` and `aa` |

### 4.3 Blink frames (3)
Periodic blink overlay, triggered every 3–5 seconds (randomised).

| ID | Description |
|----|-------------|
| `blink_open` | Eyes fully open (default — usually base portrait) |
| `blink_mid` | Eyelids half-closed |
| `blink_closed` | Eyes fully closed |

> Note: blink frames operate on the **eye region**, not the mouth region. They are a separate sprite overlay layer.

### 4.4 Idle/breathing frames (2)
Subtle resting variants for when the companion isn't speaking. Used to add life during silence.

| ID | Description |
|----|-------------|
| `idle_a` | Slight asymmetric mouth softening |
| `idle_b` | Minute lip parting (breath) |

### 4.5 Phoneme → viseme lookup table

ElevenLabs returns **character-level timestamps**, not phonemes directly. The runtime needs a two-stage lookup:

```
character (e.g. "p") → phoneme (e.g. "p") → viseme ("pp")
```

For multi-character phonemes (e.g. "th", "sh", "ch", "ng") the driver looks at character pairs.

The mapping table lives in `lib/lipsync/phoneme-viseme-map.ts`. English mapping is straightforward; **Shona-specific mappings need a small extension** for Shona-specific sounds (e.g. implosives, prenasalised consonants) — see Section 9.

## 5. Atlas generation pipeline (server-side, one-time per companion)

This runs once when a user creates or selects a new photo-based companion. Output is a single bundle stored on Supabase Storage + CDN.

### 5.1 Inputs
- **Portrait image** — user upload, or pre-curated library image, or AI-generated. Must be front-facing, neutral expression, eyes open, head roughly horizontal, mouth closed or slightly parted, clear lighting.
- **Companion ID** — to namespace the output bundle.

### 5.2 Steps
1. **Preprocess portrait**
   - Detect face landmarks (MediaPipe Face Mesh or similar). Required to:
     - Confirm portrait is usable (front-facing, single face, eyes open)
     - Extract mouth and eye region bounding boxes
     - Compute alignment transform
   - Normalise size (e.g. 1024 px tall) and orientation
   - Save as `base.png`

2. **Generate reference lip-sync video**
   - Feed `base.png` + a fixed **phoneme-rich script audio** (we record this once, in a neutral voice — call it the *atlas seed audio*) into a high-quality lip-sync model
   - Model options:
     - **MuseTalk** (self-hosted, ~$0.01/render) — v1 default
     - **Hallo3** (self-hosted, higher quality, ~$0.03/render) — v2 upgrade
     - **Sync.so API** (paid, ~$0.50–2/render) — fallback for highest-quality cases
   - Output: ~20-second 30fps reference video at 1024 px

3. **Extract viseme frames**
   - Run viseme classifier on the reference audio to label every frame with its current viseme
   - For each of the 17 mouth-related slots (12 core + 3 transitions + 2 idle), pick the single best frame from the reference video
   - "Best" = the frame whose facial landmarks most closely match the canonical shape for that viseme (cosine similarity over mouth landmark deltas)

4. **Crop mouth regions**
   - Using the face landmarks, crop a mouth box (with ~20% padding) from each chosen frame
   - Apply alpha matte (radial gradient) at edges for seamless compositing
   - Save as `mouth_<viseme_id>.png`

5. **Generate blink frames**
   - Use a simpler model (e.g. LivePortrait or a basic eye-closing affine transform) to generate three blink states
   - Crop eye region with alpha matte
   - Save as `eyes_<state>.png`

6. **Package bundle**
   - Build manifest `atlas.json` (see Section 7)
   - Zip `base.png`, all `mouth_*.png`, all `eyes_*.png`, `atlas.json`
   - Upload to Supabase Storage path `companions/<companion_id>/atlas.zip`
   - Write companion DB row with atlas URL + version

### 5.3 Infrastructure
- **Job runner**: Supabase Edge Function triggers a job on a GPU worker (RunPod / Lambda / our own A10 box)
- **Model serving**: MuseTalk container with REST endpoint. One A10 GPU handles ~3 atlas generations per minute.
- **Queue**: lightweight — Supabase table `atlas_generation_jobs` with status enum (`pending`, `running`, `done`, `failed`), polled by the worker
- **Retries**: 3 attempts on transient failures; permanent failure marks portrait as unsuitable and prompts user to upload another

### 5.4 Quality gates
Before saving the bundle:
- Each mouth crop must score ≥ a threshold on landmark match for its viseme
- Reference video must have no detected face dropouts > 100ms
- Base portrait must pass: single face, both eyes detected, mouth closed, head pose within ±15°
- Any failure → mark job failed, prompt user

### 5.5 Cost projection
| Generator | Per-atlas cost | Best for |
|---|---|---|
| MuseTalk (own GPU) | $0.01 | All users by default |
| Hallo3 (own GPU) | $0.03 | High-tier users |
| Sync API | $0.50–2 | Marketing material, special offers |

At 10k atlas generations / month using MuseTalk: ~$100/month total compute. Trivial.

## 6. Runtime engine (client-side, React Native + Expo)

### 6.1 Architecture
```
┌─────────────────────────────────────────────────┐
│  ElevenLabs Agents (or Agents-equivalent path)  │
│  - audio stream (PCM/Opus)                      │
│  - character timestamps array                   │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│  Viseme Driver (lib/lipsync/driver.ts)          │
│  - char → phoneme → viseme mapping              │
│  - schedules viseme transitions on audio clock  │
│  - emits VisemeFrameEvent at render rate (60Hz) │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│  Compositor (lib/lipsync/compositor.ts)         │
│  - loads atlas bundle (cached)                  │
│  - composites base + mouth + eyes per frame     │
│  - runs on Skia (RN Skia) or Three.js canvas    │
└─────────────────────────────────────────────────┘
```

### 6.2 The driver loop
1. ElevenLabs returns audio + `character_start_times_seconds[]` + `character_end_times_seconds[]` + `characters[]`
2. Driver collapses adjacent characters into phoneme spans (e.g. "th" → one phoneme span)
3. Maps phoneme → viseme via lookup table
4. Schedules viseme changes on the audio playback clock (not wall clock — must stay synced to audio)
5. At each render tick (60Hz): determine current viseme, determine next viseme, compute interpolation `t` for cross-fade

### 6.3 Cross-fade transitions
Between two visemes A and B over a window `T_FADE = 40ms`:
- At time `t` within the window: blend mouth_A and mouth_B with opacity `(1 - t/T_FADE)` and `(t/T_FADE)`
- This eliminates the "snap" of hard frame swaps and reads as continuous motion

### 6.4 Ambient micro-animations
Layered on top of the lip-sync composition. All deterministic-randomised, all CPU-cheap.

| Animation | Driver | Visual cost |
|---|---|---|
| **Blink** | Cron every 3–5s (randomised), 200ms total duration | Eye region sprite swap (open → mid → closed → mid → open) |
| **Head sway** | Sine wave on canvas transform, ±1.5° rotation, 4s period | Pure transform, no sprite work |
| **Breath** | Sine wave on canvas scale, ±1.5%, 3s period | Pure transform |
| **Gaze drift** | Random 2D offset on eye region, ±2 px, slow lerp | Eye sprite x/y offset |

### 6.5 Rendering backend choice
Two options, pick one for v1:

**Option A: React Native Skia** (recommended)
- Pros: native performance, mature, designed for this kind of compositing, single codebase iOS+Android
- Cons: another library in the bundle (~500 KB)

**Option B: react-native-canvas / WebGL via Three.js**
- Pros: reuses Three.js stack already in place for Rwen
- Cons: heavier, harder to debug, mobile WebGL quirks

**Decision: start with React Native Skia.** Three.js stays the 3D-character renderer. Atlas runtime stays 2D.

### 6.6 Performance budget
Per frame on a 2022 mid-range Android (Snapdragon 695):
- Atlas composite: < 4ms
- Audio playback (handled by RN Audio): < 1ms
- Driver tick: < 1ms
- Total per frame: < 6ms → comfortably 60fps headroom

Memory:
- Atlas in memory: ~5–8 MB (decoded RGBA)
- Allowable cached atlases: 2 (current companion + previous, for instant switch)

### 7.4 Supabase DB schema (additions)
Add to `companions` table (or create `companion_atlases` if separation preferred):

```sql
ALTER TABLE companions ADD COLUMN atlas_url text;
ALTER TABLE companions ADD COLUMN atlas_version int DEFAULT 1;
ALTER TABLE companions ADD COLUMN atlas_generated_at timestamptz;
ALTER TABLE companions ADD COLUMN atlas_generator text;  -- 'musetalk-v1.2' etc.
ALTER TABLE companions ADD COLUMN atlas_status text DEFAULT 'pending';  -- pending | ready | failed
```

Plus the job tracking table:

```sql
CREATE TABLE atlas_generation_jobs (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  companion_id uuid NOT NULL REFERENCES companions(id),
  status text NOT NULL DEFAULT 'pending',  -- pending | running | done | failed
  generator text NOT NULL,
  attempt int NOT NULL DEFAULT 0,
  error text,
  started_at timestamptz,
  finished_at timestamptz,
  created_at timestamptz DEFAULT now()
);
```

## 9. Open decisions & risks

### Decisions still to make
1. **Generator default**: MuseTalk vs Hallo3 for v1. MuseTalk is more mature; Hallo3 is higher quality but newer. Recommendation: ship with MuseTalk, A/B test Hallo3 in Phase 4.
2. **Worker hosting**: cloud A10 on RunPod ($0.50–0.75/hr) vs Lambda Labs ($0.60/hr) vs our own colo box. Recommendation: RunPod for v1, evaluate own box when monthly compute > $500.
3. **Atlas bundle delivery**: ship zipped, decode on device (~150ms) vs ship individual PNGs and fetch parallel. Recommendation: zipped — single CDN round-trip is faster than 20 small ones on cellular.
4. **Storage class**: keep atlases warm (Supabase Storage standard tier) vs archive after 90d inactivity. Likely keep warm; storage cost is negligible.
5. **User-uploadable portraits vs curated library only for v1**: starting with curated library reduces moderation risk and quality variance. Decision pending.

### Known risks
| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Mouth crops mismatch in lighting between visemes | Medium | Medium | Alpha-matte edges + cross-fade hides most issues. Reject portraits with harsh side lighting. |
| Atlas generation fails on uncommon face types | Medium | High | Curated library v1; gradual rollout of upload feature with quality gates |
| Audio-clock drift on Android | Low | Medium | Use ExoPlayer time API, not wall clock. Tested in Phase 2. |
| Phoneme map errors in low-resource languages (Shona) | High | Medium | Native-speaker validation pass (Phase 5) |
| MuseTalk licence/usage clarity (MIT but Tencent IP) | Low | Medium | Confirmed MIT — re-verify with internal legal before launch |
| User uploads photo of a celebrity / non-consented person | Medium | High | Defer user uploads until consent/moderation flow is built. Curated library only for v1. |

### Anti-patterns to avoid
- Don't generate atlases at user-create time synchronously (always async via job queue)
- Don't bake head motion into the mouth atlas (head sway is a separate transform layer)
- Don't hard-swap visemes without cross-fade (looks robotic)
- Don't recompute landmarks on the client every frame (atlas anchor positions are baked at generation time)
- Don't ship the atlas as part of the app bundle (per-companion downloads only — keep app slim)

### 11.1 The custom-wizard flow

Atlas generation slots into the existing 8-step wizard at
[`app/build-companion.tsx`](../../app/build-companion.tsx) (spec:
[CUSTOM-COMPANION-WIZARD-V2.md](../../docs/CUSTOM-COMPANION-WIZARD-V2.md))
as a **background job that fires at the moment the user commits their
final portrait**. The wizard never blocks on it.

```
Wizard step                             Atlas pipeline event
──────────────────────────────────────  ─────────────────────────────────────
1. Name
2. Voice (5 candidates)
3. Look (attributes + photo upload)
4. Image gen (5 portraits via Flux)
5. User picks final portrait  ────────► POST /enqueue-atlas
                                        {custom_companion_id, portrait_url}
                                        → row inserted into
                                          atlas_generation_jobs(status=pending)
                                        → returns immediately
6. Soul + Personality + Prompt          ↻ worker dequeues + runs MuseTalk
7. Idling video gen (Kling, parallel)   ↻ extracts 17 mouth frames + 3 eyes
8. Confirm + first chat                 ↻ uploads atlas.zip to Supabase Storage
                                        ↻ flips custom_companions.atlas_status
                                          to 'ready' + writes atlas_url

User arrives at chat                    one of three states:
                                        (a) atlas_status='ready'   → AtlasCompositor renders
                                        (b) atlas_status='pending' → AmbientFace fallback (Kling loop)
                                        (c) atlas_status='failed'  → AmbientFace fallback + soft retry CTA
```

**Total wizard wall time ≈ 60–90s** for the user (image gen + Kling idle
+ wizard inputs). MuseTalk atlas generation on a single A10 takes ~30s.
In practice the atlas is almost always ready by the time the user
finishes step 8 — but the architecture **never assumes it is**.

### 11.2 The `enqueue-atlas` Edge Function contract

```
POST /functions/v1/enqueue-atlas
Authorization: Bearer <user JWT>
Body:
  {
    "custom_companion_id": "uuid",
    "portrait_url":        "https://supabase.../portrait.jpg",
    "generator":           "musetalk-v1.2"        // optional, defaults to musetalk-v1.2
  }
Response:
  { "job_id": "uuid", "status": "pending" }
```

The Edge Function:
1. Validates the user owns the `custom_companion_id`
2. Validates the portrait passes preflight (single face detected, mouth
   closed, head pose within ±15°) — this is the *cheap* preflight using
   MediaPipe FaceMesh in a Deno-compatible build; the heavy work happens
   on the worker
3. Inserts the job row + sets `custom_companions.atlas_status='pending'`
4. Pings the worker via a webhook (or worker polls every 5s — TBD on
   ops side; both work)
5. Returns the job_id so the client can optionally subscribe via
   Supabase realtime for live progress

### 11.3 Client-side renderer decision

A new hook tells the chat screen which face engine to render:

```ts
// hooks/useCompanionRenderer.ts
export type Renderer = 'atlas' | 'ambient';

export function useCompanionRenderer(companionId: string): {
  renderer: Renderer;
  atlasUrl: string | null;
  loading: boolean;
} {
  // Subscribes to companion row (custom or archetype).
  // Returns 'atlas' when atlas_status='ready' AND atlas_url is set.
  // Otherwise 'ambient' — AmbientFace handles it.
}
```

Chat screen consumes it:

```tsx
const { renderer, atlasUrl } = useCompanionRenderer(activeCompanionId);
return renderer === 'atlas'
  ? <AtlasCompositor atlasUrl={atlasUrl!} audio={convAi.audioStream} timestamps={convAi.timestamps} />
  : <AmbientFace videoUrl={archetype?.idling_video_url} imageUrl={archetype?.image_url} emoji={preset?.emoji} />;
```

**The user never sees a broken state.** Atlas not ready → AmbientFace
plays the Kling idle loop. Atlas later flips to ready → next chat
session uses the new renderer.

### 11.4 Failure handling

Three failure paths, all with the same UX outcome (the companion still
works, just without the atlas):

| Failure                                     | Resolution                                          |
|---------------------------------------------|-----------------------------------------------------|
| Worker times out / crashes                  | Job retried 3× (per Section 5.3). After 3 fails →   |
|                                             | `atlas_status='failed'`, write reason to job row.   |
| Portrait fails landmark preflight           | Edge Function returns 400 *before* enqueueing.      |
|                                             | Wizard surfaces "this portrait won't lipsync well — |
|                                             | want to pick a different one of the 5?" — but does  |
|                                             | NOT block the wizard; user can still proceed and    |
|                                             | live with the AmbientFace fallback.                 |
| Atlas bundle download fails on first chat   | Client falls back to AmbientFace, retries download  |
|                                             | every minute. Cached once it arrives.               |

There's never a "no face" state. AmbientFace always works because it
falls all the way down to the preset's emoji on a coloured background.

### 11.5 Free preset companions

The 8 free presets (Rwen + Maya + Tendai + Sam + Lumi + Kai + Aria +
Zeke) and the 20+ archetype faces (Sarah, Marcus, etc.) are NOT
generated per-user — atlases for them are baked **once during launch
prep** and live in the same Storage bucket. Their DB rows ship with
`atlas_url` already set and `atlas_status='ready'`.

A script in `scripts/bake-preset-atlases.ts` runs the atlas pipeline
across every archetype where `image_url IS NOT NULL` and writes the
results back. Re-runs idempotently when we tighten the generator or
when a new archetype lands.

> Rwen is an exception. The chameleon is rendered via Three.js morph
> targets (Section 13), not the atlas system. The atlas-vs-ambient
> renderer switch checks for a `companion_kind='3d'` flag and falls
> through to the Three.js renderer for Rwen specifically.

The `atlas_generation_jobs` table from Section 7.4 covers both kinds
via a polymorphic `companion_kind text` + `companion_id uuid` (instead
of just the FK to `companions` originally shown).

### 11.7 Order of build (deferred from this turn)

When we come back to actually wire this up, the order is:

1. Migration with the DB columns above (5 min — non-blocking, all
   `not_started` defaults are safe with the AmbientFace fallback)
2. `enqueue-atlas` Edge Function with **no worker behind it yet** —
   just writes the job row. Wizard can fire it; status sits at
   `pending` forever; AmbientFace fallback handles UX. This unblocks
   the client-side work without needing GPU infra.
3. `useCompanionRenderer` hook + the AtlasCompositor stub (Skia
   canvas with empty composite, returns `'ambient'` until atlas_status
   flips to ready)
4. **Then** the GPU worker (MuseTalk on RunPod), per Phase 1 in the
   main build plan
5. Once a worker is up, run `bake-preset-atlases.ts` for the launch
   presets
6. Flip the renderer switch live — atlases that exist will render via
   AtlasCompositor; everyone else stays on AmbientFace until their
   atlas is baked

This sequencing means **we can ship the full client-side atlas plumbing
weeks before the GPU pipeline is ready**, and the day the worker lands,
every existing user's companion silently upgrades on next chat.
