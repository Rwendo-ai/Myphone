## 3. The big-three queue (next session priorities)

The user's directive on 2026-05-09 was to ship three integrated systems:

### A. XP system overhaul (~2-3 days)

Every interaction is an XP event:
- **Engagement** — login (daily), lesson completion, post create, comment,
  like, follow.
- **AI usage** — text message (lower XP), voice (mid), lipsync video
  (higher) — XP scaled to compute cost, but user pays via subscription
  not XP.
- **Purchases** — XP awarded proportional to spend (course $14.99 = X XP,
  premium $50 = 2.5×X XP).

Spend XP on: digital products (premium courses, themes, avatars). NOT on
AI usage (compute cost is real, paid via subscription).

Implementation outline:
- New table `xp_events` (user_id, event_type, amount, metadata jsonb,
  created_at). RLS owner-read, service-role-write.
- New table `xp_redemptions` (user_id, product_id, xp_spent, created_at).
- SQL function `award_xp(user_id, event_type, amount, metadata)` — inserts
  event, atomically updates `profiles.xp`.
- `lib/xp-events.ts` — client wrapper.
- Hook into AuthContext (login event), createPost, likePost, addComment,
  followUser, lesson completion, AI message paths.
- Storefront UI for spending XP on digital products.

### B. Intro bubbles / first-run tour (~1-2 days)

Contextual help bubbles that show ONCE per user per anchor:
- Triggered on first visit to a screen / first tap of a feature.
- AsyncStorage flag `intro_seen_<id>` per bubble.
- Bubble component with arrow pointing to anchor element.
- Dismissed → never shows again.

Hook locations:
- First Travel tab open → "Pick a destination from the dropdown".
- First lesson start → walk through phases (Hook, Chunks, Pattern,
  Practice, Dialogue, Recall, Mission).
- First Connections post → explain feed/like/comment/hashtag.
- First AI Companion message → text vs voice modes.

Implementation outline:
- `components/IntroBubble.tsx` (positioning, arrow, "Got it" button).
- `lib/intro-bubbles.ts` — useIntroBubble hook with AsyncStorage gate.
- `data/intro-bubbles.ts` — registry of all bubble specs (id, body, anchor
  ref hint, conditions).

### C. Paywall (Phase H — RevenueCat + IAP) (~1-2 weeks)

This is the biggest blocker for monetisation:
- RevenueCat SDK setup (already in deps, not configured).
- Products in App Store Connect + Play Console: language courses
  (consumable per-pack), Premium subscription (recurring), AI Companion
  tier upgrades.
- Purchase flow + restore purchases.
- Server-side receipt validation (RevenueCat handles most of this).
- `entitlements.ts` already exists for runtime gating; needs to read live
  RevenueCat entitlements rather than placeholder DEV_UNLOCK_ALL.
- Plans screen redesign (`/profile/plans`) with real pricing per
  jurisdiction.

This is genuinely 1-2 weeks of focused work per the original Phase H
spec in [PRODUCT-DESIGN.md](./PRODUCT-DESIGN.md). Recommended order:
RevenueCat setup → product catalogue in stores → purchase flow →
entitlements wiring → plans UI → restore + receipts → testing.

## 4. Architecture notes

### Streaming model (now applied to ALL content)

- **Lessons** (since earlier sessions) — `lessons/<courseId>/<speakerId>/<lessonId>.json`
- **Phrasebook / Culture / Safari / Flipcards** (this session) —
  `travel-content/<type>/<key>.json`
- **Audio** — `audio/{phrasebook,flipcards}/<key>/...mp3`
- **Post images** — `traveller-posts/<user_id>/<filename>.jpg`

Bundled in app: registries, types, UI, speaker pack locale JSONs (boot-
critical for i18n), companion presets, jurisdictions, crisis triggers.
Everything else streams.

### Subscription model

User runs Anthropic Max + ElevenLabs subscription. Authoring agent runs
draw from those subscriptions. We don't pay per call. If the subscription
quota hits, agents stop until the cycle resets — no surprise charges.

External APIs the user actually pays per call (DeepL, future flight
APIs, etc.) are noted explicitly; everything else assumes the
subscription model.
